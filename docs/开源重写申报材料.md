## 一、功能介绍

### 1.1 原软件（HexStrike AI）核心功能

HexStrike AI 是面向**授权渗透测试、CTF、红队演练**场景的安全自动化平台，通过 MCP 协议向大模型暴露安全工具能力。其核心能力包括：

| 类别 | 功能说明 |
|------|----------|
| **工具编排** | 通过 `hexstrike_mcp.py`（FastMCP）将 **100+** 安全工具封装为 MCP 工具，供 Agent 调用 |
| **后端服务** | `hexstrike_server.py` 提供 HTTP API（默认 `http://localhost:8888`），执行扫描、利用、信息收集等任务 |
| **场景支持** | 漏洞扫描、服务识别、Web/网络渗透辅助、CTF 解题工作流等 |
| **AI 对接** | 面向 Claude/Cursor 等 MCP 客户端；强调参数优化、日志与错误恢复 |
| **交互形态** | 以「工具执行引擎 + MCP 桥接」为主，**不内置**本地 Ollama 对话客户端与多 Agent 调度 |

原软件解决的核心问题：**让 AI 能调用专业安全工具**，但知识检索、本地模型、工具筛选与对话体验需依赖外部框架单独集成。

---

### 1.2 重写软件（HexStrike Augment）全部功能

HexStrike Augment 在保留 HexStrike 全部 MCP 工具能力的前提下，基于 **Ollama + ollmcp 客户端框架** 进行集成与扩展，形成「**对话客户端 + 多 Agent + 双路 RAG + HexStrike 工具链**」的一体化系统。

#### （1）继承自原软件（通过子模块 `mcp/`）

- HexStrike MCP 工具全集（`hexstrike-ai` 服务器）
- 与 HexStrike HTTP 后端的通信与任务执行逻辑
- 安全工具参数化调用与结果回传

#### （2）继承并增强自 ollmcp 基准客户端（`mcp_client_for_ollama/`）

- 多 MCP 服务器并行连接（stdio / SSE / HTTP）
- Ollama 本地/云端模型 tool calling、流式输出
- 工具/服务器动态启用、Human-in-the-Loop（HIL）确认
- 模型参数细调、上下文管理、性能指标展示
- 配置持久化、服务器热重载、模糊命令补全

#### （3）重写项目新增功能

| 模块 | 功能 |
|------|------|
| **自主多代理（`multi_agent/`）** | Strategy / Selector / Executor 三层架构；自然语言任务分解；工具从约 152 个筛选至约 8 个，降低上下文占用 |
| **Chroma RAG（`rag_knowledge_base/` + `rag_mcp_server.py`）** | 基于 ChromaDB + Ollama embedding 的轻量漏洞知识检索；`build_knowledge_base.py` 建库 |
| **HexStrike RAG（`hexstrike_rag/` + `hexstrike_rag_mcp_server.py`）** | 五周工程成果：BM25 + 向量 + RRF + Cross-Encoder 重排 + 规则过滤；Qdrant/Redis 可选；**`verify_payload_safety`** |
| **UltraRAG 集成（可选）** | `ultrarag_mcp_server.py` 对接 UltraRAG YAML 检索配置 |
| **统一配置与启动** | `*.example.json` 模板；`start_with_multi_agent.sh` / `.ps1` 一键启动 |
| **评估与调优工具** | `benchmark_v1.py`（Recall/nDCG/MRR）；`performance_tuner.py`（P50/P95）；`cache_layer_v2.py`（标签化缓存） |

#### MCP 工具总览（重写后）

| 服务名 | 工具示例 | 来源 |
|--------|----------|------|
| `hexstrike-ai` | 100+ 渗透/扫描类工具 | 原 HexStrike |
| `rag-knowledge-base` | `search_security_knowledge`、`get_knowledge_base_stats` | 重写（Chroma） |
| `hexstrike-rag` | 上述检索 + **`verify_payload_safety`** | 重写（增强 RAG） |

### 1.3 相较原软件的创新与优化

| 维度 | 原 HexStrike | HexStrike Augment 创新与优化 |
|------|--------------|------------------------------|
| **模型接入** | 依赖外部 MCP 客户端 | **内置 ollmcp + Ollama**，本地可离线使用 |
| **知识检索** | 无内置 RAG | **双路 RAG**：轻量 Chroma + 生产级 BM25/Qdrant 管线 |
| **检索质量** | — | 四/五步检索、重排、去重、查询扩展；Recall@10 **88%**、nDCG@10 **0.76**（见第二节） |
| **Agent 智能** | 工具直调 | **多 Agent 自主决策**，按任务动态选工具 |
| **Token 效率** | 全量工具上下文 | 筛选后单次约 **920 tokens** 工具描述，较全量 **~17,480** 节省约 **94.7%** |
| **延迟** | 取决于单次工具 | RAG 管线 P95 **185ms**（含缓存优化，见汇报材料） |
| **安全合规** | 工具执行风险提示 | 新增 **Payload 安全验证** MCP 工具 |
| **可扩展** | MCP 工具扩展 | 模块化 RAG、子模块 HexStrike、可选 UltraRAG |
| **开源协作** | 独立仓库 | 统一 `hexstrike_augment`，文档与子模块边界清晰 |

## 二、重写软件测试情况

### 2.1 测试原则

1. **不得引入恶意功能**：所有代码开源可审计；工具调用需用户显式启用；HIL 支持人工确认；`verify_payload_safety` 用于**降低误用风险**，不用于生成攻击载荷。  
2. **测试类型**：单元测试（检索/缓存/评估模块）、集成测试（MCP stdio + ollmcp）、场景测试（多 Agent + Mock RAG）。  
3. **测试环境**：Python 3.10+、Ollama、可选 Docker（Qdrant/Redis）。

### 2.2 单元测试与模块验证

| 模块 | 测试方式 | 结果摘要 |
|------|----------|----------|
| `hexstrike_rag/benchmark_v1.py` | 50/210 条标注查询，Recall@K、nDCG@K、MRR、MAP | Recall@10 **88%**，nDCG@10 **0.76**，MRR **0.83** |
| `hexstrike_rag/bm25_retriever.py` | CVE/端口/关键词分词召回 | 精确实体命中优于纯向量基线 |
| `hexstrike_rag/deduplication.py` | 精确+模糊双去重 | 去重率 **>95%**（数据层验收） |
| `hexstrike_rag/cache_layer_v2.py` | 命中率统计、标签失效 | 命中率 **60–80%**（稳态查询） |
| `hexstrike_rag/performance_tuner.py` | 分步耗时 P50/P95 | 见下表对比数据 |
| `mcp_client_for_ollama/multi_agent/` | 工具筛选、Mock 工具调用链路 | 8 工具子集可稳定加载与执行 |

### 2.3 集成测试

| 场景 | 方法 | 结果 |
|------|------|------|
| HexStrike RAG MCP × ollmcp | `HEXSTRIKE_USE_MOCK=true`，`ollmcp -j hexstrike_rag_config.json` | **15/15** 用例通过（第 5 周验收） |
| 三工具可用性 | `search_security_knowledge` / `get_knowledge_base_stats` / `verify_payload_safety` | Mock 模式下全部可调用 |
| HexStrike + 双 RAG | `hexstrike-with-rag-config.json` 三服务并联 | 客户端可发现并选择性启用工具 |
| 多代理模式 | 客户端输入 `ma`，自然语言任务 | Strategy→Selector→Executor 闭环可跑通 |

### 2.4 性能对比实验数据

以下数据来自项目阶段性汇报中的 **RAG 检索层** 优化实验（同一硬件环境、10 万条安全语料、210 条测试查询）。

#### （1）检索延迟优化

| 指标 | 优化前（第 3 周初） | 优化后（第 3–4 周） | 提升 |
|------|---------------------|---------------------|------|
| P50 延迟 | 265 ms | **95 ms** | **↓64%** |
| P95 延迟 | 420 ms | **180 ms**（验收 **185 ms**） | **↓57%** |
| 缓存命中率 | 0% | **60–80%** | 显著降低重复查询开销 |

#### （2）检索质量指标

| 指标 | 目标 | 实际 | 是否达标 |
|------|------|------|----------|
| Recall@10 | ≥85% | **88%** | ✅ |
| nDCG@10 | ≥0.75 | **0.76** | ✅ |
| MRR | ≥0.75 | **0.83** | ✅ |
| 知识库规模 | ≥100,000 条 | **100,000** | ✅ |

#### （3）多 Agent 工具上下文 Token 对比（ollmcp 客户端）

| 模式 | 工具数量（约） | 工具描述 Token（约） | 说明 |
|------|----------------|----------------------|------|
| 传统全量加载 | 152 | **17,480** | 基线：每工具约 115 tokens |
| 多 Agent 筛选后 | 8 | **920** | Selector 按任务选子集 |
| **节省比例** | — | **≈94.7%** | 支持更多轮次自主决策 |

#### （4）相对原 Chroma 单路 RAG 的扩展收益（架构层）

| 对比项 | 原 `rag_knowledge_base`（Chroma） | 新 `hexstrike_rag` |
|--------|-----------------------------------|---------------------|
| 检索方式 | 纯向量 | BM25 + 向量 + RRF + 重排 |
| 存储 | 单机 Chroma | Qdrant（可水平扩展） |
| CVE 精确匹配 | 较弱 | BM25 强化 |
| 时效/规则过滤 | 无 | 有 |
| 去重 | 无专门模块 | 双策略去重 >95% |

### 2.5 安全与合规说明

- 子模块 HexStrike 工具**仅用于授权测试**；README 与文档中已声明合规使用范围。  
- 仓库已清理历史中的 API Key、个人路径、靶场 flag 等敏感文件（`git filter-repo`）。  
- 无后门、无远程控制、无隐蔽数据外传逻辑；依赖均为公开 PyPI / 子模块。

## 三、文档说明

### 3.1 环境准备

| 依赖 | 版本/说明 |
|------|-----------|
| Python | 3.10+ |
| Git | 支持 submodule |
| Ollama | 本地服务，建议 `qwen3` 等支持 tool calling 的模型 |
| Docker | 可选，Qdrant `6333`、Redis `6379` |

### 3.2 安装步骤

```bash
# 1. 克隆（含 HexStrike 子模块）
git clone --recursive https://github.com/clutch-61/hexstrike_augment.git
cd hexstrike_augment

# 2. Python 依赖
pip install mcp rich typer httpx ollama
pip install -r mcp/requirements.txt
pip install -r hexstrike_rag_requirements.txt

# 3. 复制配置模板（将 <PROJECT_ROOT> 改为本机绝对路径）
cp hexstrike-with-rag-config.example.json hexstrike-with-rag-config.json
# Windows PowerShell:
# Copy-Item hexstrike-with-rag-config.example.json hexstrike-with-rag-config.json

# 4. 拉取 Ollama 模型
ollama pull qwen3:32b
ollama pull nomic-embed-text   # 仅构建 Chroma 知识库时需要
```

### 3.3 启动 HexStrike 后端（使用渗透工具时）

```bash
cd mcp
pip install -r requirements.txt
python hexstrike_server.py
# 服务默认 http://localhost:8888
```

### 3.4 启动 ollmcp 客户端

**Linux / macOS：**

```bash
chmod +x start_with_multi_agent.sh
./start_with_multi_agent.sh
```

**Windows：**

```powershell
.\start_with_multi_agent.ps1
```

**手动启动：**

```bash
python -m mcp_client_for_ollama.cli \
  --servers-json hexstrike-with-rag-config.json \
  --model qwen3:32b
```

### 3.5 基本使用方法

| 步骤 | 操作 |
|------|------|
| 1 | 启动后输入 `tools` 查看已连接 MCP 服务 |
| 2 | 输入 `a` 启用全部工具，或按编号选择性启用 |
| 3 | 输入 `s` 保存并返回对话 |
| 4 | 用自然语言提问，例如：「查询 CVE-2020-17519 的利用思路」 |
| 5 | 输入 **`ma`** 进入**自主多代理模式**，适合复杂渗透分析任务 |
| 6 | 输入 `hil` 可开关人工确认（执行工具前确认） |

### 3.6 构建 Chroma 知识库（可选）

```bash
python build_knowledge_base.py \
  --docs-dir ./data/extracted_readmes \
  --rebuild --test
```

### 3.7 启用 HexStrike RAG 真实管线（可选）

1. 启动 Qdrant：`docker run -d -p 6333:6333 qdrant/qdrant`  
2. 配置 `hexstrike_rag_config.json` 中 `HEXSTRIKE_USE_MOCK=false`  
3. 按 [HEXSTRIKE_INTEGRATION_GUIDE.md](../HEXSTRIKE_INTEGRATION_GUIDE.md) 导入语料
